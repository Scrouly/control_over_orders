# Deep audit report: `control_over_orders`

Дата: 2026-03-03

## Методика
- Проведён статический обзор ключевых модулей (`views`, `forms`, `models`, `settings`, `management commands`, `tests`).
- Выполнена проверка Python-синтаксиса через `py_compile`.
- Для рисков, завязанных на рантайм/БД, зафиксированы условия воспроизведения.

## Критичные и высокие проблемы

### 1) Несогласованность обязательности `controller` (модель vs формы/импорт)
**Что найдено:**
- В модели `Assignment.controller` поле обязательное (нет `null=True`, `blank=True`).
- В `AssignmentCreateForm` и `AssignmentForm` контролирующий допускается как необязательный.
- В `assignment_create` в БД передаётся `controller=data.get('controller')`, т.е. потенциально `None`.
- В импорте DBF `controller` тоже может быть `None`, но дальше идёт безусловный `Assignment.objects.create(...)`.

**Риск:** `IntegrityError` при создании поручений (через UI и/или DBF-импорт), возможно падение всего процесса импорта внутри `transaction.atomic()`.

**Рекомендация:** выбрать единый контракт:
1. либо сделать `controller` nullable в модели + миграция,
2. либо сделать поле обязательным в формах/импорте и добавить валидацию/понятные ошибки до сохранения.

---

### 2) Потенциальный массовый rollback импорта из DBF
**Что найдено:**
- Импорт выполняется внутри одного `transaction.atomic()` на весь цикл.
- При проблеме в любой строке (например, `controller=None` при NOT NULL) откатится весь прогон.

**Риск:** потеря времени и невозможность частичного успешного импорта; одна «плохая» запись ломает всю партию.

**Рекомендация:**
- либо перейти на батчи,
- либо отлавливать ошибки по записи, логировать и продолжать,
- либо валидировать запись до `create`.

---

### 3) Широкие `except Exception` в пользовательских потоках
**Что найдено:** в `assignments/views.py` несколько операций уведомлений обёрнуты в `except Exception` и переводятся в message-строку.

**Риск:** реальные баги (например, ошибки импорта, неверные параметры, БД-исключения) маскируются как «ошибка отправки», без разделения по классу и без структурного логирования.

**Рекомендация:**
- ловить узкие классы исключений,
- добавлять `logger.exception(...)` для трассировки,
- для критичных исключений отдавать корректный код/алерт.

## Средние проблемы

### 4) Необработанный `json.loads(request.body)` в AJAX-endpoints
**Что найдено:** update/delete endpoints справочников читают JSON body без `try/except json.JSONDecodeError`.

**Риск:** невалидный JSON даст HTTP 500 вместо ожидаемого 400 и ломает UX/контракт API.

**Рекомендация:** добавить безопасный parse helper с возвратом `JsonResponse({'error': ...}, status=400)`.

---

### 5) Генерация номера документа не атомарна
**Что найдено:** `next_document_number` вычисляет номер на основе «последней записи», но сам endpoint только предлагает номер; далее при параллельном создании возможны одинаковые номера.

**Риск:** коллизии в бизнес-идентификаторе, особенно при активной многопользовательской работе.

**Рекомендация:**
- либо ввести уникальный constraint (с учётом бизнес-правил),
- либо формировать номер в транзакции/сервисе при реальном сохранении,
- либо использовать выделенный sequence table.

---

### 6) Жёстко прошитый URL в bulk-редиректе
**Что найдено:** при действии `print` используется строка `'/reports/print-selected/?ids=...'` вместо `reverse('reports:print_selected')`.

**Риск:** ломкость при изменении маршрутов или префиксов.

**Рекомендация:** заменить на `reverse`.

---

### 7) Дублирование бизнес-логики обновления просроченных
**Что найдено:** одинаковая логика обновления `OVERDUE` есть и во view (`check_overdue_view`), и в management command (`check_overdue`).

**Риск:** рассинхрон при будущих изменениях критериев.

**Рекомендация:** вынести в единый сервис/функцию доменного слоя и переиспользовать.

## Низкие/операционные проблемы

### 8) `runbot` не валидирует токен до старта
**Что найдено:** создаётся `Bot(token=settings.TELEGRAM_BOT_TOKEN)` без явной проверки наличия токена.

**Риск:** непонятные падения в окружениях с неверной конфигурацией.

**Рекомендация:** ранняя валидация + понятное сообщение в stderr.

---

### 9) Тесты появились, но покрытие критичных бизнес-веток всё ещё ограничено
**Что найдено:** добавлены smoke/unit тесты, но нет плотного покрытия для:
- DBF-импорта (валидные/невалидные строки, частичные ошибки),
- сценариев уведомлений по всем типам,
- правок справочников с невалидным JSON,
- конкурентного создания поручений/номеров.

**Рекомендация:** добавить интеграционные тесты уровня business flow и контракты на ошибки API.
