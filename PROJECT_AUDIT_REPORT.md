# Отчёт по анализу проекта `control_over_orders`

Дата: 2026-03-03

## Краткое резюме
В проекте обнаружены несколько архитектурных и эксплуатационных недочётов: один критичный (дублирование полей модели), несколько средних (устойчивость/обслуживаемость), и системный пробел в тестах.

## Найденные недочёты

### 1) Критично: дублирование полей в модели `Assignment`
**Наблюдение:** поля `status`, `created_at`, `updated_at` объявлены в `Assignment` дважды.

**Риск:** неочевидное поведение метакласса Django, путаница в миграциях/ревью, высокий шанс регрессий при доработках модели.

**Где:** `task_control/models.py`.

**Рекомендация:** оставить только одно объявление каждого поля, сформировать миграцию, проверить обратную совместимость admin/form/queryset-кода.

---

### 2) Средне: обработка ошибок в Telegram-уведомлениях через `print` и `except Exception`
**Наблюдение:** в отправке уведомлений используются `print(...)` и широкие перехваты исключений.

**Риск:** сложнее мониторить инциденты, нельзя централизованно собирать логи и алерты, трудно диагностировать частичные отказы.

**Где:** `telegram/notifications.py` (`send_telegram_message` и блоки отправки).

**Рекомендация:** перейти на `logging` (структурированные сообщения), сузить перехват исключений, добавить retry/backoff для сетевых ошибок.

---

### 3) Средне: небезопасный `ALLOWED_HOSTS` при `DEBUG=True`
**Наблюдение:** при включённом `DEBUG` разрешаются все хосты (`['*']`).

**Риск:** в неверно настроенной среде (особенно staging) повышается площадь атаки Host Header и риск некорректных редиректов/ссылок.

**Где:** `config/settings.py`.

**Рекомендация:** вынести `ALLOWED_HOSTS` в env, не использовать wildcard даже для debug-окружения на сервере.

---

### 4) Средне: фильтр ФИО в `deadline_filter_view` реализован через объединение QuerySet
**Наблюдение:** поиск по исполнителю строится как `qs.filter(...last_name...) | qs.filter(...first_name...)`.

**Риск:** менее читаемо, сложнее расширять (middle name, нормализация), возможны дубликаты/неочевидный SQL-план на больших выборках.

**Где:** `reports/views.py`.

**Рекомендация:** заменить на `Q(executor__last_name__icontains=...) | Q(executor__first_name__icontains=...)` в одном `.filter(...)`.

---

### 5) Средне: низкое покрытие тестами по приложениям
**Наблюдение:** в `core/tests.py`, `task_control/tests.py`, `references/tests.py`, `reports/tests.py`, `telegram/tests.py` — только заглушки.

**Риск:** регрессии в критичных user-flow (логин/роли, отчёты, справочники, уведомления) не будут ловиться автоматически.

**Где:** тестовые файлы приложений.

**Рекомендация:** приоритизировать smoke+integration тесты:
1. доступы/роли (`staff_required`, admin-only операции),
2. ключевые POST-ветки в `references`/`reports`,
3. уведомления (моки HTTP/Telegram),
4. отчётные фильтры и сортировки.

---

### 6) Низко: заглушечный модуль `telegram/views.py`
**Наблюдение:** модуль содержит только стандартный шаблон `# Create your views here.` и не используется в URL routing.

**Риск:** технический шум, ложные ожидания о web-интерфейсе Telegram-приложения.

**Где:** `telegram/views.py`, `config/urls.py`.

**Рекомендация:** либо удалить модуль/приложение из web-части, либо добавить явную документацию, что приложение используется только как слой моделей/уведомлений/management-команд.

## Приоритизированный план работ
1. **P0 (срочно):** починить модель `Assignment` (дубли полей) + миграции + регрессионный прогон.
2. **P1:** улучшить observability уведомлений (logging, узкие исключения, retry).
3. **P1:** расширить тестовое покрытие критичных потоков (минимум smoke suite по 5 приложениям).
4. **P2:** рефактор фильтра ФИО в отчётах на `Q`-условия.
5. **P3:** почистить/задокументировать неиспользуемые каркасы (`telegram/views.py`).

## Что проверено в рамках анализа
- Ручной просмотр ключевых файлов (`models`, `views`, `settings`, `tests`, `urls`).
- Базовая синтаксическая проверка Python-файлов через `py_compile`.
